## 1. Context & Background

Third-party regulatory compliance audits require absolute certainty that event logs have not been manipulated or selectively filtered before or during conformance checking. In traditional architectures, audit logs are stored in databases where administrators or attackers could retroactively alter event sequences to hide process violations (e.g., bypassing a required segregation-of-duty check).

To establish an immutable audit trail, the **Cryptographic Receipt Authority (CRA)** provides tamper-proof validation of process replay proofs. When a log is replayed against a process model (such as a POWL model or a Petri Net), the sequence of state transitions is cryptographically chained using the BLAKE3 hash function. The final resulting state is signed by the execution runtime, producing an immutable **Receipt**. This receipt proves to external auditors that a specific sequence of events occurred, in a specific order, and was replayed against a specific, unmodified process model, without exposing the raw underlying event data.

## 2. Technical Specifications & Boundaries

### 2.1 Cryptographic Receipt Structure

The receipt is represented as a structured ledger payload. We define the `Receipt` struct in `crates/c8-receipts/src/lib.rs`:

```rust
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Debug, Clone, PartialEq, Eq)]
pub struct Receipt {
    /// SHA3-256 or BLAKE3 hash of the target reference model (Petri Net / POWL).
    pub model_id: [u8; 32],
    /// The final chained hash resulting from the BLAKE3 trace-event progression.
    pub final_hash_chain: [u8; 32],
    /// Cryptographic signature of the validator authority verifying compliance.
    pub authority_signature: [u8; 64],
    /// The final verdict (Perfect, NonConforming, Deadlocked).
    pub verdict: ConformanceVerdict,
    /// Timestamp of execution.
    pub timestamp_epoch: u64,
}

#[derive(Serialize, Deserialize, Debug, Clone, Copy, PartialEq, Eq)]
pub enum ConformanceVerdict {
    PerfectAlignment,
    FitnessDeficit,
    DeadlockEncountered,
}
```

### 2.2 BLAKE3 Hash Chaining Mechanism

During trace replay, the state is accumulated step-by-step. The hash of each step depends on the hash of the previous step, the event payload, and the specific transition fired in the model:

$$H_i = \text{BLAKE3}(H_{i-1} \mathbin{\Vert} \text{EventID}_i \mathbin{\Vert} \text{TransitionID}_i \mathbin{\Vert} \text{TokenMarking}_i)$$

```rust
use blake3::Hasher;

pub struct TraceChainer {
    hasher: Hasher,
}

impl TraceChainer {
    pub fn new() -> Self {
        let mut hasher = Hasher::new();
        hasher.update(b"W4PM_RECEIPT_SALT_V1");
        Self { hasher }
    }

    /// Accumulate a single step into the cryptographic hash chain.
    pub fn chain_step(&mut self, event_id: &[u8], transition_id: &[u8], marking_state: &[u8]) {
        self.hasher.update(event_id);
        self.hasher.update(transition_id);
        self.hasher.update(marking_state);
    }

    /// Finalize the hash chain.
    pub fn finalize(self) -> [u8; 32] {
        let hash = self.hasher.finalize();
        hash.into()
    }
}
```

### 2.3 Compile-Time Witness signature & Boundary Gating

To prevent developers from bypassing the receipt authority or exporting conformance statements without producing a valid proof, the API enforces type-level witness checks.
Any function attempting to export a conformance boundary without a proving witness must fail compilation, or be explicitly marked as unsafe and fail safety checks.

## 3. Acceptance Criteria

1. **BLAKE3 Integrity Checks**:
   - Unit tests must replay a trace, calculate the `final_hash_chain`, modify a single event ID in the middle of the history, replay it again, and assert that the resulting hash is completely different.
2. **Trybuild Compile-Fail verification**:
   - A trybuild test under `tests/ui/` must attempt to compile a module that calls `export_boundary_no_witness` with the `strict` feature enabled, and verify that the compiler fails with a clean, descriptive message about missing witness proofs.
3. **Receipt Serialization**:
   - Receipts must serialize and deserialize to JSON and Bincode without loss of data, maintaining exact parity with the expected hashes.
4. **Implementation Boundaries**:
   - File size must be at least 1500 bytes.
   - The code must be compliant with corporate style guidelines regardless of keyword presence.
