## 1. Context & Background

Surfacing compilation diagnostics is only the first step in aiding developer productivity within the process-evidence framework. When developers encounter a type-law refusal (for example, attempting to flat-map a multi-perspective event log without specifying the handling of dropped dimensions, or using unadmitted raw event structs inside a safe sandbox block), the type system rejects the code. Resolving these compile errors manually requires writing highly specific boilerplate code, introducing import statements, or inserting error conversion handlers.

The **`unLSP` Code Actions Engine** automates the resolution of these compiler diagnostics. By responding to `textDocument/codeAction` requests, the server provides the developer with instant, contextual quick-fixes. These quick-fixes automatically restructure code to satisfy compile-time laws (e.g., wrapping raw evidence in `Admit::admit()`, attaching the correct `LossPolicy` enum, or generating witness proofs). This significantly reduces the learning curve of the nightly Rust process-evidence system and enforces correct compliance conventions.

## 2. Technical Specifications & Boundaries

### 2.1 Code Action Request Processing

The server intercepts `textDocument/codeAction` JSON-RPC payloads. It maps the incoming `Diagnostic` code and target range to a list of available `CodeAction` options.

```rust
use tower_lsp::jsonrpc::Result;
use tower_lsp::lsp_types::*;
use std::collections::HashMap;

pub async fn handle_code_actions(
    uri: Url,
    diagnostics: Vec<Diagnostic>,
) -> Result<Option<CodeActionResponse>> {
    let mut actions = Vec::new();

    for diag in diagnostics {
        if let Some(NumberOrString::String(ref code)) = diag.code {
            match code.as_str() {
                // W4PM-002: Format Covenant Violation (Lossy projection missing LossPolicy)
                "W4PM-002" => {
                    actions.push(create_loss_policy_quickfix(&uri, &diag));
                }
                // W4PM-001: Refusal Law Violation (Raw event admission missing)
                "W4PM-001" => {
                    actions.push(create_admission_quickfix(&uri, &diag));
                }
                // W4PM-007-ERR-01: Missing Witness implementation
                "W4PM-007-ERR-01" => {
                    actions.push(create_witness_quickfix(&uri, &diag));
                }
                _ => {}
            }
        }
    }

    if actions.is_empty() {
        Ok(None)
    } else {
        Ok(Some(actions))
    }
}
```

### 2.2 Quick-Fix Transformations

#### A. Format Covenant Quick-Fix (`LossPolicy`)
When a lossy operation (such as `flatten()`, `project()`, or `flatten_ocel_to_xes()`) is called without a `LossPolicy` argument, the server proposes inserting a default policy.
- **Before**: `let xes_log = project(ocel_log);`
- **After**: `let xes_log = project(ocel_log, LossPolicy::KeepAll);`

#### B. Admission Boundary Quick-Fix (`Admit::admit`)
If a raw, unvalidated variable enters a safe process block, it triggers a refusal. The quick-fix inserts the `Admit::admit()` call to gate-check the variable.
- **Before**: `let safe_event = raw_event;`
- **After**: `let safe_event = wasm4pm_compat::admission::Admit::admit(raw_event).map_err(|e| e.into())?;`

#### C. Witness Annotation Quick-Fix
If an event stream is exported without a witness proof, the quick-fix generates a local witness token block or attributes.
- **Insertion**:
  ```rust
  struct TraceWitness;
  impl wasm4pm_compat::law::Witness for TraceWitness {
      type Model = ReferenceModel;
  }
  ```

### 2.3 WorkspaceEdit Construct

Edits are packed into a `WorkspaceEdit` containing document changes:

```rust
fn create_admission_quickfix(uri: &Url, diag: &Diagnostic) -> CodeActionOrCommand {
    let mut changes = HashMap::new();
    let replacement = "wasm4pm_compat::admission::Admit::admit(raw_event).map_err(|e| e.into())?".to_string();
    
    changes.insert(
        uri.clone(),
        vec![TextEdit {
            range: diag.range,
            new_text: replacement,
        }],
    );

    CodeActionOrCommand::CodeAction(CodeAction {
        title: "Admission Law: Admit raw variable into safe boundaries".to_string(),
        kind: Some(CodeActionKind::QUICKFIX),
        diagnostics: Some(vec![diag.clone()]),
        edit: Some(WorkspaceEdit {
            changes: Some(changes),
            ..Default::default()
        }),
        is_preferred: Some(true),
        ..Default::default()
    })
}
```

## 3. Acceptance Criteria

1. **JSON-RPC Compliance**:
   - The handler must return a valid `CodeActionResponse` containing the `WorkspaceEdit` objects with precise ranges matching the diagnostic locations.
2. **Transform Correction Tests**:
   - Integration tests must run the code action handler on actual Rust code snippets containing typical errors (e.g. missing `LossPolicy`), apply the returned text edits, and assert that the resulting file:
     - Parses as valid Rust.
     - No longer generates the corresponding compiler diagnostics.
3. **No Code Corruption**:
   - The text edit ranges must be calculated relative to UTF-8 offsets to prevent code mangling in editors with different line-ending styles (LF vs CRLF).
4. **Implementation Boundaries**:
   - File size must be at least 1500 bytes.
   - The code must be compliant with corporate style guidelines regardless of keyword presence.

### 4. Implementation Details, Code Snippets & Delta Descriptions

#### A. Specific Implementation Instructions
1. Establish a new module `code_actions` inside the `src/lsp` namespace.
2. Define the main entry point `handle_code_actions` accepting a `Url` for the document and a list of active `Diagnostic` values.
3. Iterate over the diagnostics to match the custom process-evidence error codes: `W4PM-001` (refusal law boundary check) and `W4PM-002` (format covenant checks).
4. Implement `create_admission_quickfix` which construct a valid `WorkspaceEdit` containing the required replacement code wrapper: `wasm4pm_compat::admission::Admit::admit(raw_event).map_err(|e| e.into())?`.
5. Return the collection of generated quick-fixes packaged inside a tower-lsp compatible `CodeActionResponse`.

#### B. Definition of Done (DoD)
1. The code action resolver compiles cleanly on the nightly toolchain.
2. Code actions return valid `WorkspaceEdit` actions with ranges mapping exactly to diagnostic coordinates.
3. Edits generated by the quick-fix are syntactically valid Rust.
4. Calculations of offsets are verified against UTF-8 boundaries to prevent text corruption in files containing non-ASCII characters or CRLF line endings.
5. All generated test cases run successfully and verify that applying the edits resolves the corresponding diagnostic error.
