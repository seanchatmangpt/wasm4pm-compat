## 1. Context & Background

In continuous production process monitoring, alignment and conformance scores must be tracked in real-time. As business processes evolve or operational deviations occur, execution logs drift from their reference models. This phenomenon, known as **process drift**, can be sudden (e.g., caused by an immediate system change or policy update) or gradual (e.g., caused by seasonal variation or progressive operator bypasses).

A key challenge is the continuous, low-overhead computation of conformance deltas between the reference models and the live event streams. To maintain mathematical integrity, the quality metrics—specifically **fitness** and **precision**—must be strictly bounded to the interval `[0, 1]` at the type level. Allowing out-of-bounds metrics (e.g., fitness greater than 1.0 or division-by-zero errors) in the monitoring stream would invalidate downstream automated auditing engines and generate false compliance alerts.

The **Parity Delta Analyzer** serves as the core evaluation component that:
1. Replays event streams against reference models.
2. Identifies specific deviation categories (log-only and model-only moves).
3. Evaluates conformance drift boundaries.
4. Generates structured delta reports while leveraging `Between01` const generics to guarantee mathematical correctness at compile time.

## 2. Technical Specifications & Boundaries

### 2.1 Conformance Delta Representation

The analyzer must calculate and report the exact deviations between the observed log and the process model. We define the following types in `src/parity/delta.rs`:

```rust
use wasm4pm_compat::law::Between01;

/// Represents the classification of a replay step.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum MoveCategory {
    /// The event occurred in the execution log but could not be matched to any enabled transition in the model.
    LogOnly,
    /// The model required a transition to fire to progress, but no corresponding event was present in the log.
    ModelOnly,
    /// The log event and the model transition fired in synchronization.
    SyncMove,
}

/// A structured report summarizing the conformance delta of a trace.
#[derive(Debug, Clone, PartialEq)]
pub struct DeltaReport<const FIT_NUM: u64, const FIT_DEN: u64, const PREC_NUM: u64, const PREC_DEN: u64>
where
    Between01<FIT_NUM, FIT_DEN>: /* inhabited */,
    Between01<PREC_NUM, PREC_DEN>: /* inhabited */,
{
    pub total_log_only_moves: u64,
    pub total_model_only_moves: u64,
    pub total_sync_moves: u64,
    pub fitness: FitnessConst<FIT_NUM, FIT_DEN>,
    pub precision: PrecisionConst<PREC_NUM, PREC_DEN>,
}
```

### 2.2 Mathematical Safety Bounds via `Between01`

We enforce conformance metric correctness using the `Between01` const-generic system. Any attempt to initialize a `DeltaReport` with metrics that violate the interval `[0, 1]` (i.e., where `NUM > DEN` or `DEN == 0`) must fail compilation.

```rust
use wasm4pm_compat::law::{FitnessConst, PrecisionConst};

impl<const FIT_NUM: u64, const FIT_DEN: u64, const PREC_NUM: u64, const PREC_DEN: u64>
    DeltaReport<FIT_NUM, FIT_DEN, PREC_NUM, PREC_DEN>
where
    Between01<FIT_NUM, FIT_DEN>: /* inhabited */,
    Between01<PREC_NUM, PREC_DEN>: /* inhabited */,
{
    /// Construct a new DeltaReport, ensuring at compile time that metrics lie in [0, 1].
    pub fn new(log_only: u64, model_only: u64, sync: u64) -> Self {
        Self {
            total_log_only_moves: log_only,
            total_model_only_moves: model_only,
            total_sync_moves: sync,
            fitness: FitnessConst::<FIT_NUM, FIT_DEN>::new(),
            precision: PrecisionConst::<PREC_NUM, PREC_DEN>::new(),
        }
    }
}
```

### 2.3 Drift Boundary Triggers

The system monitors metric changes over sliding trace windows. A drift alert is triggered when the rolling fitness drops below a target threshold:

```rust
pub struct DriftMonitor<const THRESH_NUM: u64, const THRESH_DEN: u64>
where
    Between01<THRESH_NUM, THRESH_DEN>: /* inhabited */,
{
    threshold: FitnessConst<THRESH_NUM, THRESH_DEN>,
    history: Vec<f64>,
}
```

## 3. Acceptance Criteria

1. **Compilation Guarantee**:
   - `DeltaReport::<4, 5, 9, 10>::new(2, 1, 15)` must compile successfully because $4/5 \le 1.0$ and $9/10 \le 1.0$.
2. **Trybuild Compile-Fail Verification**:
   - A trybuild test must assert that attempting to construct `FitnessConst<3, 2>` or `DeltaReport::<3, 2, 1, 1>::new(...)` fails compilation with:
     `the trait bound 'Require<false>: IsTrue' is not satisfied`.
   - The trybuild test suite must execute cleanly under `tests/ui/` during local validation.
3. **Structured Formatting**:
   - The `DeltaReport` must implement `std::fmt::Display` to serialize conformance stats into a standardized JSON or human-readable format.
4. **Implementation Boundaries**:
   - File size must be at least 1500 bytes.
       - The code must contain no stubs, mock implementations, or TODO comments.
