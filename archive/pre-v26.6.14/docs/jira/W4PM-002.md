# JIRA Ticket: W4PM-002
## Title: Formalize OCEL 2.0 Tuple Validation and Implement Named Law Refusal Types

### 1. Context & Background
Object-Centric Event Logs (OCEL) 2.0 introduce multi-perspective structural complexity that cannot be validated using flat table schemas. Relying on lax dynamic schema validation leads to silent data corruption, missing event-to-object relations, and structural drift that causes conformance checker execution to fail unpredictably.

According to the formal definition by Ghahfarokhi et al., an OCEL 2.0 log is defined as a tuple containing:
- **E**: A set of Events.
- **O**: A set of Objects.
- **EA**: Event Attributes mapping.
- **OA**: Object Attributes mapping.
- **E2O**: Event-to-Object relationships mapping.
- **O2O**: Object-to-Object relationships mapping.

To ensure that downstream runtimes operate on correct structures, we must formalize this tuple validation at the type boundary. When projecting or flattening the multi-perspective log into a single-perspective trace (e.g. for traditional XES-based analysis), any loss of relations or attributes must be governed by an explicit policy. A silent flattening process is a major source of compliance errors.

---

### 2. Technical Specifications & Boundaries

#### A. OCEL 2.0 Tuple Parser
The parser must reconstruct the formal tuple at ingestion. The parser validates the integrity of relations (such as preventing dangling object references or invalid O2O links) before admitting the data.

```rust
pub struct OcelTuple<E, O, EA, OA, E2O, O2O> {
    events: Vec<E>,
    objects: Vec<O>,
    event_attrs: Vec<EA>,
    object_attrs: Vec<OA>,
    event_object_relations: Vec<E2O>,
    object_object_relations: Vec<O2O>,
}
```

#### B. Projection, `LossPolicy`, and `LossReport`
Flattening a multi-perspective process model or log involves projecting the log onto a subset of object types. We define the `Project` trait:

```rust
pub trait Project<TargetDimension> {
    type Output;
    type Error;

    fn project(
        &self, 
        policy: LossPolicy
    ) -> Result<(Self::Output, LossReport), Self::Error>;
}
```

The `LossPolicy` defines what attributes or relationships can be safely omitted:

```rust
pub enum LossPolicy {
    StrictNoLoss,
    AllowAttributeLoss,
    AllowRelationshipLoss,
    Custom(fn(&LossReport) -> bool),
}

pub struct LossReport {
    pub dropped_event_relations: usize,
    pub dropped_object_relations: usize,
    pub omitted_attributes: Vec<String>,
}
```

If the projection incurs loss not explicitly allowed by the policy, it must refuse the transition.

#### C. Named Law Refusal Types
We discard generic error enums. Instead, we implement specific, struct-based named law refusal types as part of the public API:

```rust
pub struct MissingObjectRelation {
    pub event_id: String,
    pub expected_object_id: String,
    pub relation_type: String,
}

pub struct FlatteningLoss {
    pub policy_violated: LossPolicy,
    pub report: LossReport,
}
```

These types must implement a common `Refusal` trait that logs diagnostic data without leaking unadmitted raw log content.

#### D. Trybuild Test Target: `flattening_without_policy.rs`
To verify compilation enforcement, we must write a trybuild integration test `tests/ui/flattening_without_policy.rs`. This test attempts to compile a projection call without supplying a `LossPolicy`. The compiler must reject this with a clean error message requiring a policy.

2. The `Project` trait is implemented and returns a `LossReport` describing any dropped relations.
3. Only named law refusal types (`MissingObjectRelation`, `FlatteningLoss`) are used to report validation or projection failures.
4. The trybuild test `flattening_without_policy.rs` compiles successfully and verifies compilation failure when no policy is supplied.
