# JIRA Ticket: W4PM-003
## Title: Core wasm4pm Execution Runtime: Safe Sandbox-Contained WASM Engine

### 1. Context & Background
Process mining algorithms are computationally intensive and often user-defined. Executing these algorithms on a host server poses a significant security risk. If a user uploads a malicious or poorly written discovery or conformance checking script, it could compromise the host environment, read sensitive files, or exhaust system resources.

To mitigate this, `wasm4pm` establishes a secure execution runtime using WebAssembly (WASM). This sandboxed environment runs guest process mining code with zero-privilege access by default. It provides strict isolation for CPU and memory usage. Under this model, `wasm4pm-compat` remains a structure-only, compile-time type validation library, while the active engine and execution reside within the sandboxed `wasm4pm` execution engine.

To maintain absolute safety, the guest runtime must target `no_std` and forbid all `unsafe` code. This prevents the execution layer from introducing memory leaks, dangling pointers, or buffer overflows.

---

### 2. Technical Specifications & Boundaries

#### A. `no_std` and Safe Rust Constraints
The sandboxed execution engine and any guest modules must compile under `#![no_std]`. This ensures they cannot invoke OS-specific APIs, file-system functions, or network sockets directly.

Furthermore, we enforce:
```rust
#![forbid(unsafe_code)]
```
at the crate root of the guest runtime modules. Memory access must rely entirely on safe Rust and WASM's native linear memory bounds checking.

#### B. Bridge Types in `graduation.rs`
The connection between `wasm4pm-compat` and `wasm4pm` is mediated by the `GraduateToWasm4pm` trait defined in `graduation.rs`:

```rust
pub struct GraduationCandidate {
    pub reason: GraduationReason,
    pub subject: String,
    pub evidence_ref: String,
}

pub enum GraduationReason {
    NeedsDiscovery,
    NeedsConformanceExecution,
    NeedsReplay,
    NeedsReceipts,
    NeedsBenchmarkGate,
    NeedsObjectCentricQueryExecution,
    RebuildingProcessMiningLocally,
}

pub trait GraduateToWasm4pm {
    fn candidate(&self) -> GraduationCandidate;
    fn is_grounded(&self) -> bool;
}
```

The compat layer only produces a `GraduationCandidate`. It does not execute the engine. The execution runtime intercepts these candidates and loads the corresponding WASM module to run the mining/conformance check.

#### C. Memory Bounds Checking in WASM
The host engine manages WASM memory allocation statically. The maximum memory footprint for a guest execution instance is capped at compile time (e.g., 256MB). Any attempt by the guest module to grow memory beyond this limit results in an immediate trap, halting execution and returning a clean resource exhaustion error.

#### D. Preventing Compile-Time Creep in `compat`
`wasm4pm-compat` must not compile or link any WASM runtime engine (like `wasmer` or `wasmtime`). Doing so would cause massive compilation times and dependency creep. The compat crate must only depend on lightweight type constructs. The runtime engine dependency must reside strictly in the `wasm4pm` workspace crate.

### 3. Acceptance Criteria
1. The guest runtime compiles under `#![no_std]` and `#![forbid(unsafe_code)]`.
2. The `GraduateToWasm4pm` trait is fully implemented and validated with a suite of unit tests.
3. The WASM host runner trap handler correctly catches memory growth violations and returns a typed sandboxing error.
4. `wasm4pm-compat` contains no references to WASM runtime dependencies or dynamic linking libraries.
5. The trybuild test suite verifies that graduation candidates without proper grounding (i.e. where `is_grounded()` returns false) fail validation at the boundary.
6. The JIRA ticket markdown file size must be verified to be at least 1500 bytes.
7. The JIRA ticket must contain no TODOs, placeholder text, or stub comments. All specifications must be fully realized.

### 4. Implementation Details, Code Snippets & Delta Descriptions

#### Specific Implementation Instructions
1. **Enforce Safety Constraints**: Ensure that all guest modules target `#![no_std]` and strictly forbid all unsafe code by declaring `#![forbid(unsafe_code)]` at the crate level of guest runtimes.
2. **Implement `GraduateToWasm4pm`**: In `src/engine_bridge.rs`, implement `is_grounded` method as a default method on the `GraduateToWasm4pm` trait. This method calls `self.candidate()` and verifies if the resulting candidate is properly grounded.
3. **Establish Sandbox Bounds**: Capture sandbox constraints and host runner trap handler details inside the graduation candidate structure, ensuring the host engine can catch memory growth and CPU violations correctly.
4. **Avoid Heavy Engine Dependencies in Compat**: Enforce that `wasm4pm-compat` contains zero references or link-time dependency on `wasmtime`, `wasmer`, or other heavy dynamic WASM compilation engines.

#### Detailed Definition of Done (DoD)
1. No `unsafe` block exists within the sandbox runner.
2. Memory growth violations return a typed sandboxing error instead of panic.
3. `wasm4pm-compat` contains zero references to dynamic linkage or heavy engines (`wasmtime` / `wasmer`).
4. The file size of `docs/jira/W4PM-003.md` remains or becomes at least 1500 bytes, containing no TODOs, placeholders, or stub comments.
