# JIRA Ticket: W4PM-005
## Title: Parity Test Harness: Multi-Perspective OCEL Oracle and Parity Fixtures

### 1. Context & Background
As we transition from traditional Python/OCPA (Object-Centric Process Analysis) reference libraries to our native Rust-based `wasm4pm` runtime, we must ensure absolute behavioral parity. Any deviation in fitness scores, precision calculations, or alignment paths will invalidate conformance verdicts and lead to inconsistent reports.

To guarantee that the Rust `wasm4pm` runtime behaves identically to the reference implementation, we require a comprehensive Parity Test Harness. This harness reads structured JSON oracle outputs (generated from Python/OCPA) and compares them against the results produced by the Rust engine. The verification must run on complex multi-perspective datasets containing multiple object types, event-to-object relations, and non-trivial workflow nets.

---

### 2. Technical Specifications & Boundaries

#### A. JSON Oracle Input/Output Format
The test harness must read parity fixtures from a standard location (e.g., `tests/fixtures/parity/`). The JSON file contains both input logs/models and the expected outputs:

```json
{
  "input_ocel_path": "tests/fixtures/logs/order_to_cash.json",
  "expected_fitness": 0.984251,
  "expected_precision": 0.912048,
  "expected_alignments": [
    {
      "case_id": "case_001",
      "alignment_path": ["receive_order", "check_inventory", "ship_goods"],
      "cost": 0
    }
  ]
}
```

#### B. Floating-Point Epsilon Validation
Due to differences in floating-point operations between Python (using 64-bit floats under CPython) and Rust (`f64`), direct equality assertions will fail. The harness must enforce a strict epsilon tolerance of $10^{-6}$:

```rust
pub struct ParityComparer;

impl ParityComparer {
    pub fn assert_epsilon_close(actual: f64, expected: f64) {
        let diff = (actual - expected).abs();
        assert!(
            diff < 1e-6,
            "Parity violation: actual {}, expected {}, diff {} (exceeds epsilon 1e-6)",
            actual,
            expected,
            diff
        );
    }
}
```

#### C. Multi-Perspective Process Model Validator
For multi-perspective models, the validator checks synchronized execution paths across different object types. It ensures that when an event is executed, it updates the state vectors for all associated object types in the exact order specified by the reference oracle.

#### D. Process Cube Dimension Verification (`ProcessCubeLaw`)
The process cube module must conform to the `ProcessCubeLaw` (refer to `docs/PROCESS_CUBE_LAW.md`). The test harness must verify that slicing a process cube along a given dimension produces the exact same sub-log and subsequent fitness metrics as the reference implementation:

```rust
pub struct ProcessCubeLaw {
    pub dimension: String,
    pub slice_value: String,
    pub expected_sublog_hash: [u8; 32],
}
```

Any discrepancy in cell extraction or dimension indexing must trigger a parity test failure.
