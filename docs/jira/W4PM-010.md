## 1. Context & Background

When a process alignment check or a token replay operation determines that an execution log is non-conforming, simply outputting a low fitness score is insufficient for compliance debugging or automated remediation. Auditors and software developers need a detailed, precise breakdown of where the process execution broke down. This is particularly critical in complex systems (such as high-frequency trading or distributed supply chains) where a single out-of-order event can ripple downstream, creating deadlocks or leaving orphan tokens in the process space.

The **Residual Failset Analyzer** is responsible for parsing non-conforming execution traces to isolate the exact structural points of failure. By evaluating the residual token distribution (marking) after a replay fails, the analyzer constructs a structured **`ResidualFailset`**. It identifies both:
1. **Unreplayable Transitions**: Activities in the log that were attempted but were not enabled by the current marking.
2. **Deadlocking States**: States where tokens are trapped in places with no outgoing enabled transitions, preventing the process from reaching a valid final state.

The analyzer translates these execution errors into named type-system refusals (e.g. `UnreplayableClaim`), mapping them to standard compiler `Diagnostic` traits. This allows the same failure analysis to feed both developers' IDEs (via LSP) and external compliance reporting engines.

## 2. Technical Specifications & Boundaries

### 2.1 The `ResidualFailset` Type

We define the primary failset structures in `src/verifier/failset.rs`:

```rust
use serde::{Serialize, Deserialize};
use std::collections::HashMap;

/// Captures the complete breakdown of a failed conformance replay.
#[derive(Serialize, Deserialize, Debug, Clone, PartialEq)]
pub struct ResidualFailset {
    /// List of events in the log that requested transitions not enabled by the model marking.
    pub unreplayable_transitions: Vec<FailedTransition>,
    /// Place identifiers that still contain tokens, but have no enabled transitions.
    pub deadlocked_places: Vec<String>,
    /// The final remaining token marking of the model.
    pub residual_marking: HashMap<String, u32>,
}

#[derive(Serialize, Deserialize, Debug, Clone, PartialEq, Eq)]
pub struct FailedTransition {
    /// The unique identifier of the event from the execution log.
    pub event_id: String,
    /// The activity name associated with the event.
    pub activity: String,
    /// The list of missing predecessor places required to enable this transition.
    pub missing_preplaces: Vec<String>,
}
```

### 2.2 Diagnostic Mapping & Refusals

Replay failures map directly to the `UnreplayableClaim` refusal. To integrate with reporting tools and LSP diagnostics, the analyzer implements a `DiagnosticReport` trait:

```rust
pub trait ComplianceDiagnostic {
    /// Convert the compliance failure into a structured diagnostic report.
    fn to_diagnostic(&self) -> DiagnosticPayload;
}

#[derive(Serialize, Deserialize, Debug, Clone)]
pub struct DiagnosticPayload {
    pub code: String,
    pub severity: String,
    pub message: String,
    pub offending_event_id: Option<String>,
    pub metadata: HashMap<String, String>,
}

impl ComplianceDiagnostic for FailedTransition {
    fn to_diagnostic(&self) -> DiagnosticPayload {
        let mut metadata = HashMap::new();
        metadata.insert("activity".to_string(), self.activity.clone());
        metadata.insert("missing_preplaces".to_string(), self.missing_preplaces.join(","));

        DiagnosticPayload {
            code: "W4PM-010-FAIL-UNREPLAYABLE".to_string(),
            severity: "ERROR".to_string(),
            message: format!(
                "UnreplayableClaim: Activity '{}' (Event: {}) could not be fired. Missing preplaces: {:?}",
                self.activity, self.event_id, self.missing_preplaces
            ),
            offending_event_id: Some(self.event_id.clone()),
            metadata,
        }
    }
}
```

### 2.3 Verification Report Output

The analyzer outputs a comprehensive `VerificationReport` which combines the cryptographic receipt (if any sync replay succeeded) or the residual failset (if compliance failed):

```rust
#[derive(Serialize, Deserialize, Debug, Clone)]
pub struct VerificationReport {
    pub file_path: String,
    pub trace_id: String,
    pub is_compliant: bool,
    pub failset: Option<ResidualFailset>,
}
```

## 3. Acceptance Criteria

1. **Analyzer Trace Verification**:
   - Unit tests must feed the analyzer a pre-defined non-conforming trace (e.g., executing a "Ship Product" activity before a "Receive Payment" activity in a purchase-to-pay model). The test must assert that:
     - The output `VerificationReport` has `is_compliant = false`.
     - The `failset` contains exactly one `FailedTransition` listing "Ship Product".
     - The `missing_preplaces` field contains the place name representing "Payment Received".
2. **Diagnostic Trait Correctness**:
   - The generated `DiagnosticPayload` must contain the exact code `W4PM-010-FAIL-UNREPLAYABLE` and have all metadata keys correctly populated.
3. **No Deadlock Left Unreported**:
   - If the replay terminates with remaining tokens that cannot move (deadlock), the analyzer must populate `deadlocked_places` with the non-empty place names.
4. **Implementation Boundaries**:
   - File size must be at least 1500 bytes.
   - The code must be compliant with corporate style guidelines regardless of keyword presence.
