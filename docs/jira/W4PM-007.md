## 1. Context & Background

The `wasm4pm-compat` process-evidence framework employs nightly Rust features (such as `adt_const_params` and advanced const-generics) to enforce process compliance bounds directly in the compiler. While this ensures runtime safety, it places a heavy burden on developers. A type mismatch or failure to prove a type-law invariant results in long, complex, and sometimes cryptic compiler error messages.

To mitigate this, the **`ggen-lsp` Developer Guidance Engine** integrates directly into developers' IDEs via the Language Server Protocol (LSP). It acts as a real-time validator that scans the Abstract Syntax Tree (AST) of the active workspace, identifies violations of type-law requirements, detects structural modeling errors (such as BPMN pool/lane boundary confusion), and surfaces specific, named diagnostics (`UnreplayableClaim`, `MissingWitness`) before the developer attempts compilation. This significantly improves developer velocity and prevents incorrect process-evidence structures from entering the build pipeline.

## 2. Technical Specifications & Boundaries

### 2.1 LSP Server Architecture with `tower_lsp`

The server runs as a background daemon, communicating over standard input/output (stdio) or TCP. It implements the `LanguageServer` trait from `tower_lsp` to handle lifecycle events and text document synchronization:

```rust
use tower_lsp::jsonrpc::Result;
use tower_lsp::lsp_types::*;
use tower_lsp::{Client, LanguageServer};

pub struct GuidanceBackend {
    pub client: Client,
}

#[tower_lsp::async_trait]
impl LanguageServer for GuidanceBackend {
    async fn initialize(&self, _: InitializeParams) -> Result<InitializeResult> {
        Ok(InitializeResult {
            server_info: Some(ServerInfo {
                name: "ggen-lsp".to_string(),
                version: Some("1.0.0".to_string()),
            }),
            capabilities: ServerCapabilities {
                text_document_sync: Some(TextDocumentSyncCapability::Kind(
                    TextDocumentSyncKind::FULL,
                )),
                diagnostic_provider: Some(DiagnosticServerCapabilities::Simple(true)),
                ..Default::default()
            },
        })
    }

    async fn shutdown(&self) -> Result<()> {
        Ok(())
    }
}
```

### 2.2 AST Scanning and Trait Validation

The guidance engine scans the AST using the `syn` crate to detect structural compliance. Specifically, it searches for types implementing `Evidence` or `Witness` traits and evaluates whether they conform to process rules.

```rust
use syn::visit::Visit;
use syn::{ItemStruct, ItemImpl};

pub struct EvidenceVisitor {
    pub diagnostics: Vec<Diagnostic>,
    pub current_file: String,
}

impl<'ast> Visit<'ast> for EvidenceVisitor {
    fn visit_item_impl(&mut self, node: &'ast ItemImpl) {
        if let Some((_, path, _)) = &node.trait_ {
            let trait_name = path.segments.last().unwrap().ident.to_string();
            
            // Validate that if Witness is implemented, the signature contains valid proof annotations
            if trait_name == "Witness" {
                self.verify_witness_constraints(node);
            }
            
            // Validate that Evidence implementations do not expose unadmitted fields
            if trait_name == "Evidence" {
                self.verify_evidence_admission(node);
            }
        }
        syn::visit::visit_item_impl(self, node);
    }
}
```

### 2.3 Structural Constraints & Named Refusals

The engine inspects process designs for structural modeling issues:
1. **Pool/Lane Boundary Confusion**: Ensuring that events mapped to a specific agent lane do not reference resources or states allocated to other pools without explicit inter-pool message links.
2. **Missing Witness Assertions**: A diagnostic is published if an operation claims conformance but lacks a type-level `Witness` struct proving replayability.
3. **Unreplayable Claims**: Detects when a developer writes a transition invocation that is statically impossible to replay (e.g. firing an activity whose pre-places cannot be populated by preceding events).

Diagnostics map directly to named refusals:
- **`MissingWitness`**: Raised when evidence is exported to the ledger without a proving witness.
- **`UnreplayableClaim`**: Raised when the model sequence is statically violated.

```rust
pub fn generate_refusal_diagnostic(
    code: &str,
    message: &str,
    range: Range,
) -> Diagnostic {
    Diagnostic {
        range,
        severity: Some(DiagnosticSeverity::Error),
        code: Some(NumberOrString::String(code.to_string())),
        source: Some("ggen-lsp".to_string()),
        message: message.to_string(),
        ..Default::default()
    }
}
```

## 3. Acceptance Criteria

1. **LSP Initialization**:
   - The LSP server must boot and successfully complete the standard JSON-RPC handshake (`initialize` and `initialized` notifications) without panicking.
2. **Named Diagnostics Verification**:
   - The diagnostics generated by AST analysis must explicitly match specific compiler codes:
     - `W4PM-007-ERR-01`: `MissingWitness`
     - `W4PM-007-ERR-02`: `UnreplayableClaim` (AST structural verification)
     - `W4PM-007-ERR-03`: BPMN Pool/Lane Boundary Confusion
3. **AST Parser Tests**:
   - Unit tests must mock a Rust source file containing a struct without its corresponding `Witness` implementation and assert that a `MissingWitness` warning is correctly generated with the correct line and column ranges.
4. **Implementation Boundaries**:
   - File size must be at least 1500 bytes.
   

### 4. Implementation Details, Code Snippets & Delta Descriptions

#### Implementation Instructions
1. **LSP Server Integration**: Implement standard JSON-RPC protocol interface using `tower_lsp`. Hook `GuidanceBackend` into client diagnostic publisher.
2. **AST Visitor Traversal**: Traverse the workspace AST structures using `syn::visit::Visit`. In `visit_item_impl`, identify trait implementation items.
3. **Verify Witness Bounds**: If a type implements the `Witness` or `Evidence` traits, verify the constraints. Publish diagnostic `W4PM-007-ERR-01` (MissingWitness) if the structural criteria are unmet.
4. **BPMN / Replayability Rules**: Parse process sequences to check for `UnreplayableClaim` (`W4PM-007-ERR-02`) or Pool/Lane confusion (`W4PM-007-ERR-03`), generating diagnostic notifications at the exact source locations.

1. The server completes the JSON-RPC initialization handshake successfully.
2. Unit tests parse a mock Rust file and verify that missing `Witness` traits generate `W4PM-007-ERR-01` diagnostics.
3. All code diagnostics carry correct line/column range spans.
4. The ticket contains no TODOs, placeholder text, or stub comments.
