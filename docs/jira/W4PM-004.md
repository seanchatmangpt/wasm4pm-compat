# JIRA Ticket: W4PM-004
## Title: SIMD-Accelerated Conformance Kernels: Token Replay and Streaming DFG

### 1. Context & Background
In industrial process monitoring, systems must analyze large-scale streams of process events in real time. Standard single-threaded token replay or alignment algorithms cannot sustain the high throughput required for sub-millisecond conformance verdicts on complex multi-perspective logs. 

To achieve the necessary performance, we must exploit SIMD (Single Instruction, Multiple Data) data parallelism and sliding event windows. Replaying tokens across multiple independent traces or computing Directly-Follows Graph (DFG) transitions can be vectorized. This ticket covers the implementation of conformance checking kernels designed to leverage `portable_simd` and custom heuristics to optimize alignment searches.

---

### 2. Technical Specifications & Boundaries

#### A. SIMD-Accelerated Token Replay via `portable_simd`
The conformance engine must use Rust nightly's `core::simd` (portable_simd feature) to perform parallel state updates on Petri net marking vectors. For a net with up to 64 places, markings can be stored in a `u32x16` or `u64x8` SIMD vector:

```rust
use std::simd::num::SimdUint;
use std::simd::u32x16;

pub struct SimdMarking {
    pub vector: u32x16,
}

impl SimdMarking {
    pub fn fire_transitions(&mut self, input_mask: u32x16, output_mask: u32x16) {
        // Subtract token requirement (inputs) and add tokens (outputs) in a single instruction
        self.vector = (self.vector - input_mask) + output_mask;
    }
}
```

This vectorization allows the engine to evaluate 16 place transitions simultaneously, increasing throughput for complex workflow nets.

#### B. Sliding Event Window: `EventWindow<T, SIZE>`
For streaming DFG construction, we introduce a sliding window structure that maintains event counts and transition frequencies:

```rust
pub struct EventWindow<T, const SIZE: usize> {
    buffer: [Option<T>; SIZE],
    head: usize,
    count: usize,
}

impl<T, const SIZE: usize> EventWindow<T, SIZE> {
    pub fn push(&mut self, event: T) -> Option<T> {
        // Monotonically advances window, returning the evicted event
        let old = self.buffer[self.head].take();
        self.buffer[self.head] = Some(event);
        self.head = (self.head + 1) % SIZE;
        old
    }
}
```

This sliding window must enforce strict temporal invariants. If an event is pushed with a timestamp earlier than a preceding event in the window, it violates the monotonic time law.

#### C. Rejecting Temporal Order Confusion
We must explicitly prevent the mixing of online streaming events with offline batch events that can lead to out-of-order logs. We define a named refusal type:

```rust
pub struct TemporalOrderConfusion {
    pub current_timestamp: u64,
    pub offending_timestamp: u64,
}
```

If `offending_timestamp < current_timestamp`, the event is refused at the boundary and not admitted into the streaming DFG.

#### D. A* Alignment Search Optimization
To calculate fitness and precision scores, the engine must compute alignments (the shortest path matching log traces to model paths). We implement an optimized A* search algorithm:
- The heuristic function estimates the remaining cost using state-space reduction techniques.
- Open set operations utilize a binary heap prioritized by estimated total cost.
- SIMD is used to calculate the heuristic distance vector for all active nodes in parallel.

#### Specific Implementation Instructions
1. **Leverage SIMD Vectors**: Implement parallel state updates on Petri net marking vectors using nightly Rust's `core::simd` features (specifically `u32x16` or `u64x8`).
2. **Implement Sliding Event Window**: Create a sliding window event queue `EventWindow` that maintains a monotonic timestamp order for events. Out-of-order events pushed to the window should trigger verification failure.
3. **Define Temporal Order Confusion Refusal**: Define a named refusal type `TemporalOrderConfusion` containing the current and offending timestamps to reject out-of-order events at the boundary.
4. **Vectorize A* Search Heuristics**: Parallelize A* alignment distance estimation using SIMD registers to execute conformance alignments in sub-millisecond execution times.
