# JIRA Ticket: W4PM-001
## Title: Architect Type-Safe Refutable Lifecycle Transitions and Admission Boundaries in wasm4pm-compat

### 1. Context & Background
In high-integrity process mining runtimes, the ingestion of untrusted log inputs presents a major safety and correctness challenge. Allowing unvalidated, malformed, or out-of-order process data to leak into downstream conformance runtimes can cause panic states, memory errors, or incorrect analytical verdicts.

The wasm4pm-compat architecture gates untrusted inputs directly at the type boundary. This prevents unadmitted or malformed process data from escaping into downstream execution layers. Instead of checking constraints at runtime via ad-hoc assertions, the system enforces that all applications conform upward to the nightly Rust type-law substrate. 

By leveraging Rust's type system, the compilation phase itself serves as the primary verification gate. The type signature of a function dictates its evidence requirements. Unvalidated process logs cannot be passed to algorithms expecting admitted logs because they represent different, incompatible types. This boundary is designed to be completely non-invertible, preventing any downward degradation.

---

### 2. Technical Specifications & Boundaries

#### A. Sealed Trait `EvidenceState`
We must implement a sealed trait pattern to prevent downstream crates from introducing custom, invalid lifecycle states:

```rust
pub mod sealed {
    pub trait Sealed {}
}

pub trait EvidenceState: sealed::Sealed {
    type Token;
    const IS_ADMITTED: bool;
}
```

By placing the `Sealed` trait in a private module, we guarantee that only the states defined in `wasm4pm-compat` can implement `EvidenceState`.

#### B. Mutually Exclusive Types: `AdmittedEvidence<T>` and `RefusedEvidence<T>`
We define two distinct wrappers around process evidence of type `T`:

```rust
pub struct AdmittedEvidence<T> {
    data: T,
    hash: [u8; 32],
}

pub struct RefusedEvidence<T> {
    raw_data: T,
    refusal_reason: String,
    code: u32,
}
```

These types must be mutually exclusive. We construct them through non-invertible state transitions.

#### C. Non-Invertible State Transitions
Evidence must flow through a strictly monotonic state pipeline:
`Raw` -> `Admitted` -> `Exportable` -> `Receipted`

Transitions can only proceed forward. Once evidence is marked `Admitted`, it cannot be converted back to `Raw` or downgraded. Each transition requires satisfying structural invariants, backed by const generic constraints.

```rust
pub struct Raw;
pub struct Admitted;
pub struct Exportable;
pub struct Receipted;

impl sealed::Sealed for Raw {}
impl sealed::Sealed for Admitted {}
impl sealed::Sealed for Exportable {}
impl sealed::Sealed for Receipted {}

impl EvidenceState for Raw {
    type Token = ();
    const IS_ADMITTED: bool = false;
}

impl EvidenceState for Admitted {
    type Token = [u8; 32];
    const IS_ADMITTED: bool = true;
}
```

#### D. Generic Const Expressions for Place/Transition Size Boundaries
To enforce memory and performance bounds at compile-time, we leverage `generic_const_exprs`. The maximum size of a workflow net (places and transitions) must be bounded:

```rust
pub struct BoundedWfNet<const PLACES: usize, const TRANSITIONS: usize>
where
    [(); PLACES + TRANSITIONS]:,
{
    // Net components
}

impl<const PLACES: usize, const TRANSITIONS: usize> BoundedWfNet<PLACES, TRANSITIONS>
where
    [(); PLACES + TRANSITIONS]:,
    Assert<{ PLACES + TRANSITIONS <= 4096 }>: IsTrue,
{
    pub fn new() -> Self {
        Self {}
    }
}

pub enum Assert<const COND: bool> {}
pub trait IsTrue {}
impl IsTrue for Assert<true> {}
```

### 4. Implementation Details, Code Snippets & Delta Descriptions

#### Specific Implementation Instructions
1. **Restructure `EvidenceState`**: Modify `src/state.rs` by moving the `Sealed` trait out of the private `private` module and into a public `sealed` module. The `EvidenceState` trait will be updated to include an associated type `Token` and a boolean constant `IS_ADMITTED`.
2. **Implement Canonical Typestate Tokens**: For each typestate token (`Raw`, `Parsed`, `Admitted`, `Refused`, `Projected`, `Exportable`, `Receipted`), implement both `sealed::Sealed` and `EvidenceState`. Specifically:
   - For `Raw`, `Parsed`, `Refused`, `Projected`, `Exportable`, and `Receipted`, `Token` is set to `()` and `IS_ADMITTED` is set to `false`.
   - For `Admitted`, `Token` is set to `[u8; 32]` and `IS_ADMITTED` is set to `true`.
3. **Introduce wrappers `AdmittedEvidence<T>` and `RefusedEvidence<T>`**: In `src/evidence.rs`, define `AdmittedEvidence<T>` containing `pub data: T` and `pub hash: [u8; 32]`, and `RefusedEvidence<T>` containing `pub raw_data: T`, `pub refusal_reason: String`, and `pub code: u32`.
4. **Const-Generic Bounded Workflow Net**: In `src/petri.rs`, implement `BoundedWfNet` using const generics `const PLACES: usize` and `const TRANSITIONS: usize`. It verifies that the sum of places and transitions is at most 4096.
