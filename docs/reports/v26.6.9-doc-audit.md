### 2.3 Stale References to `ts` and `wasm` Features
The public features of `wasm4pm-compat` are exactly `formats`, `strict`, and `wasm4pm`. The features `ts` and `wasm` were deleted from `Cargo.toml` in previous updates. However, they are still referenced in several places in the codebase and test scripts, causing warnings or failing checks.

1. **`tests/graduation.rs`**:
   * **Line 47**: `#[cfg(feature = "ts")] mod ts_tests { ... }`
   * **Line 74**: `#[cfg(all(feature = "wasm", target_arch = "wasm32"))] mod wasm_tests { ... }`
   * *Status:* Stale. These test blocks are inactive because the features do not exist, triggering compiler check-cfg warnings (`unexpected_cfgs`). The code blocks (`ts` bindings generation and `wasm` boundary function validations) are dead code.
2. **`src/ts/` and `src/wasm/` modules**:
   * These directories exist and contain full Rust sources (`src/ts/mod.rs`, `src/wasm/mod.rs` etc.), but their parent module declarations are not present in `src/lib.rs`.
3. **`ggen/audits/audit-feature-isolation.sh`**:
   * The feature isolation audit script checks for `ts` and `wasm` features explicitly.
   * **Rule 3** (lines 215-228) and **Rule 4** (lines 233-255) check if `ts` and `wasm` modules are gated by `#[cfg(feature = "ts/wasm")]` in `lib.rs`, which fails (exit code 1) because the modules have been removed from `lib.rs` entirely.
   * Lines 387-413 check for `serde`, `specta`, `tsify`, and `wasm-bindgen` as optional dependencies in `Cargo.toml`. Because these dependencies were removed alongside the features, the grep checks fail.
4. **`ggen/audits/AUDIT_SPEC.md`**:
   * Lines 26, 33, and 58 assert that `ts` and `wasm` are gated features.
5. **Other emitted metadata referencing obsolete features**:
   * `emitted/environment-hooks.md` (lines 59, 62, 188, 189)
   * `emitted/hardening/dto-classification.md` (lines 57, 63, 89)
   * `emitted/plugin-registry-systems.md` (lines 53, 56, 286, 287)

### 2.4 Generated and Source-Caste Language
To comply with the "fruit-after-kind" covenant, rendered source files are first-class peers (no `generated/` directory prefix and no `DO NOT EDIT` banners). Several files still contain comment headers referring to themselves as "generated," which violates this doctrine:

* **`src/lib.rs:223`**: `/// Generated by `ggen sync --rule witness-markers`; maintained via TTL + ggen, not by hand.`
* **`src/ts/export.rs:150`**: `"/* Generated compile-time TypeScript type definitions (WASM Boundary Law) */\n\n{}"`
* **`tests/graduation.rs:63`**: `// Optionally write generated TypeScript bindings to external paths if configured.`
* **`DAY3_WASM4PM_COMPAT_TO_WASM4PM_YIELD.md:66`**: Quotes the stale comment: `/// Generated by `ggen sync --rule witness-markers`...`

### 2.5 Engine Behavior Claims Inside Compat
`wasm4pm-compat` is structure-only. The audit confirmed:
* **No engine imports** exist in the main library `src/` directory.
* Engine functions (`simulate_replay`, `compute_alignment`, `discover_model`, etc.) are absent.
* Examples do not simulate execution.
* Active documents (`README.md`, `docs/GRADUATION.md`, `docs/DESIGN_DECISIONS.md`) correctly state thatcompat does not execute process-mining algorithms and that computation graduates to `wasm4pm`.

One minor documentation issue was found:
* **`docs/GRADUATION.md:5`**: `Graduation lives behind the wasm4pm feature (src/graduation.rs).`
  * *Status:* Stale. There is no `src/graduation.rs`. The graduation types and traits reside in `src/engine_bridge.rs`.
