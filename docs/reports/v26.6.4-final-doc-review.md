## 2. Core Archetype & Architectural Distinction

To prevent conceptual leaks, the documentation set maintains a clean and unambiguous separation between the following four ecosystem components:

1.  **`wasm4pm-compat`**: The lightweight, nightly-always Rust library defining zero-cost compile-time structural invariants, state-transition typestates, and witness metadata constraints for process evidence. It contains **no execution logic, replay solvers, or conformance algorithms**.
2.  **`ggen`**: The Ostar generative pipeline tool and ontology compiler. It compiles manifests and turtle ontologies (e.g. `wasm4pm-compat.ttl` defining standard witnesses) into concrete Rust structures (e.g. `src/witnesses.rs`) and test fixtures. It does not run at application runtime.
3.  **`wasm4pm`**: The active execution engine and conformance checking runtime. It is responsible for heavy dynamic computations, including token game replay, state-space reachability expansions, conformance check alignment solver execution, and predictive outcome calculations.
4.  **`unrdf daemon`**: The metadata storage query background service that manages graph-based ontology databases and evaluates SPARQL check hooks.

## 7. Adversarial Challenge Report (Critic)

### Challenge 1: Evasion of the Admission Boundary
-   **Assumption**: Downstream callers cannot instantiate an `Admitted` evidence wrapper without validating the data.
-   **Attack Scenario**: A malicious developer tries to directly construct `Evidence { value, state: PhantomData::<Admitted>, witness: PhantomData::<W> }` to bypass the admission checks.
-   **Verdict**: **BLOCKED**. The `Evidence` struct's internal fields (specifically `value`, `state`, and `witness`) are private/crate-restricted. Outside the crate, the only public methods to obtain an `Admitted` evidence are transitions consuming a valid `Admission<T, W>`. Since the constructor for `Admission` is private, it can only be minted by the `Admit` trait.
-   **Blast Radius**: Zero.

### Challenge 2: Evasion of the Loss Policy Boundary
-   **Assumption**: Silent information loss is prevented during projection.
-   **Attack Scenario**: A caller performs a conversion using `Project::project` but ignores the returned `LossReport` or tries to project without supplying a policy.
-   **Verdict**: **BLOCKED**. The signature of `Project::project` consumes `self` and returning a `Result<Evidence<Target, Projected, W>, Refusal>`. If the caller passes `LossPolicy::RefuseLoss`, the compiler evaluates if any item is dropped, failing the operation at runtime with a typed `Refusal` if any elements were lost. If they pass `AllowLossWithReport`, they are forced to handle the `LossReport` type, keeping the process auditable.
